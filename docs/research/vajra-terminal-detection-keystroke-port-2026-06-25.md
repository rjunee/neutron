## TL;DR — the substrate gap

Vajra's tmux era treated the **pane as the I/O channel**: it scanned `capture-pane` text for state signatures and reacted with `send-keys`. Over ~14 months it accreted ~21 distinct detectors, each encoding a specific production incident.

Neutron's PTY substrate deliberately **moved turn I/O off the terminal** — answers flow out-of-band through the dev-channel MCP `reply` → HTTP `/reply` sink. tmux is gone; the PTY is read through a single 16 KB rolling ring buffer and written through one raw `child.write()` seam. As a result:

- **Read API:** one private closure `ring` (16 KB, `RING_BUFFER_MAX`) in `persistent-repl-substrate.ts:945`, exposed read-only *only* under `NEUTRON_REPL_DEBUG=1` via `session.debugRing()`. **There is no general "recent terminal text" accessor and no line/bottom-N addressing.**
- **Write API:** `PtyChild.write(data)` (`pty-host.ts:27`) — raw string only, `\r` for Enter. **There is no structured send-keys helper** (no Escape / Ctrl-C / arrow keys / numbered-option / multi-key sequence).
- **Output-signature detectors that exist:** exactly **ONE** — the dev-channel first-run disclaimer auto-dismiss. Everything else (wedge, post-spawn, heartbeat, crash) keys off **process liveness (`hasExited`/`kill -0`) + HTTP `/health` + transcript JSONL**, never PTY text.
- Trust/bypass dialogs are **PRESENT but via disk pre-seed** (`ensure-claude-trust.ts` writes `hasTrustDialogAccepted`/`bypassPermissionsModeAccepted` to `.claude.json`) so those Ink dialogs never render — no keystroke needed.

**Consequence:** Every Vajra detector that worked by *scanning pane text for a signature and injecting a response* is **MISSING in Neutron** except the disclaimer case. Critically, the interactive prompts CC emits **regardless of I/O channel** — tool-use approval, compact-resume picker, `/rate-limit-options` org-cap menu, and the `AskUserQuestion` multi-select — STILL render as Ink TUI prompts in the PTY and **will still deadlock the REPL** with no keystroke path. These are the genuine hard-won lessons to port.

## Cross-cutting invariants the port MUST preserve

These are transport-independent lessons; carry them into the PTY substrate regardless of which detector you port:

1. **Edge-triggered, latched alerting — never level/time-dedupe.** Latch on absent→present, clear only on present→absent. (Mechanisms 3, 10, 11. The hourly-re-fire-on-stale-banner bug.)
2. **Doc-quote guards on every content match.** Reject signatures inside inline-backticks, fenced blocks, markdown bullets, or `+`/digit-led diff lines. Anchor cursor matches at line start (`^❯`). (Mechanisms 1, 4, 10, 11.)
3. **Bottom-N positional guards.** Match footers/menus only within the bottom N lines of the viewport; a status panel can render *below* the footer (24-line window, learned the hard way). (Mechanisms 1, 4.)
4. **Stamp the debounce BEFORE the await.** A transport-level write failure must NOT retry next tick and risk a double-keystroke on an approval prompt. (Mechanisms 2, 4.) In Neutron: idempotent reads may retry; **mutating `writeKey`/`write` must be fire-once per detection.**
5. **JSONL / disk is the source of truth for liveness — port probes lie.** `/health` returned OK while turns wedged for minutes. Read the transcript and ask the OS (`kill -0`, `lsof`, `hasExited`) at decision time, not cached state. (Mechanisms 8, 11, 18, 19, 20.)
6. **Per-thread respawn caps with hard-stop + auto-clear-on-health.** Cooldown (10min) + hard cap (3/hr) → `capped_at` HARD STOP + ONE operator alert; auto-clear when the topic answers a real turn. A re-arming cause loops forever without it. (Mechanisms 8, 14, 19.)
7. **Re-read AFTER the readiness gate flips.** A pre-health pane snapshot is stale; re-capture fresh after `/health` up before acting. (Mechanism 6.)
8. **First-ready grace window.** 60s grace before wedge detection trusts a "lost signal"; each respawn re-stamps it (else a legitimate 30-90s boot trips a false wedge). (Neutron already carries this in `topic-readiness-tracker` parlance — preserve on every new detector.)
9. **Bounded, single-threaded output processing.** Vajra learned that fanned-out `capture-pane` collapsed the event loop (PR #84, #90). The PTY ring is already single-callback — keep detector passes O(ring) and off the hot write path; never fan out blocking process spawns inside a tick.

## Recommended PORT SEQUENCE

**Phase 0 — foundations (blocking; nothing content-based ports without these):**
1. **F1** ring-read accessor (`getRecentOutput` + normalized matching + bottom-N).
2. **F2** structured keystroke API (`writeKey`: enter/escape/ctrl-c/up/down/digit).
3. **F3** output-scan tick scaffold (edge-latch + doc-quote + bottom-N + before-await debounce helpers).

**Phase 1 — P0 + P1 (daily-driver blockers; CC emits these regardless of I/O channel):**
4. **#1 Wedged-interactive-prompt detect+recover** — the dispatch-locked flagship; AskUserQuestion/menu deadlock. (Needs F1+F2+F3.)
5. **#2 Auto-approve tool-use prompt** — key-to-kingdom approvals block silently.
6. **#3 Compact-resume picker** — resume-after-compact deadlock.
7. **#4 /rate-limit-options org-cap auto-stop.**
8. **#8 Stuck-turn JSONL watchdog** — the "port probe lied, turn wedged" class; biggest reliability win.
9. **#11 API-5xx dead-turn notifier** (JSONL watcher) — mid-turn 5xx eats the reply.
