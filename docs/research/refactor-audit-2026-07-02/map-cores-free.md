# Subsystem map: `cores/free/` — the bundled Tier-1 free Cores

Audit date: 2026-07-02. All paths relative to `/Users/ryan/repos/neutron-open`.

## 1. Purpose & responsibilities

`cores/free/` hosts the nine bundled first-party product-feature plugins ("Cores") that ship with every Neutron Open install: **tasks, reminders, calendar, email (managed), research, code-gen, scraping, google-workspace, agent-settings**. (A tenth, **notes**, was removed in PR #161 — `git log cores/free/notes` — but a stray `node_modules/` remains.) Each Core is an "npm-shape" workspace package: a `package.json` with a Zod-validated `"neutron"` manifest block (capabilities, MCP tool schemas, UI components, secrets), an `index.ts` barrel, and a `src/` tree. At gateway boot, `cores/runtime/bundled-registry.ts` walks `cores/` (one container level deep — `bundled-registry.ts:189`, `loader.ts:231-268`), validates every manifest (a bad bundled manifest is a **blocking boot error**, `bundled-registry.ts:42-47`), and `gateway/cores/install-bundled.ts:138` auto-installs them: it dynamically imports each barrel and duck-types `buildTools(deps)` (required), `buildExtraTools(deps)` (optional), and a `LAUNCHER_ICON` UI module export (`install-bundled.ts:751-784, 854`). A Core surfaces (a) MCP tools gated by `CapabilityGuard.wrapToolHandler` + `SecretAuditLog`, (b) a slash chat-command (`/task`, `/remind`, `/cal`, `/email`, `/research`, `/scrape`, `/code`), and (c) declarative launcher-tile / app-tab metadata.

## 3. Public seams / contracts other subsystems consume

1. **`package.json.neutron` manifest** — parsed by `@neutronai/cores-sdk` `parseManifest` (`cores/sdk/manifest.ts`); walked at boot by `cores/runtime/bundled-registry.ts` + `loader.ts`. Manifest invalid ⇒ gateway refuses to start.
2. **Barrel duck-type contract** — `gateway/cores/install-bundled.ts:751-784` dynamically imports `cores/free/<x>/index.ts` and requires `buildTools(deps)`; optionally merges `buildExtraTools(deps)` (`:854`) and reads `LAUNCHER_ICON = {emoji,label}` from the manifest-declared UI entry-point module (`:231, :289-294`). **This is the real runtime contract and it is documented nowhere in the SDK** (see §6.3).
3. **`CoreBackendFactoryMap`** — `gateway/boot-helpers.ts:745 buildCoresBackendFactories` builds one backend per slug; threaded via `composition.cores.backends` by `gateway/cores/mount-open-cores.ts:280` (Open) and the carved-out Managed composer.
4. **Chat-command filters** — first-match-wins chain (`gateway/boot-helpers.ts:519 buildChainedChatCommandFilter`); Open chain order calendar → email → reminders → research (`mount-open-cores.ts:310-329`).
5. **Barrel exports consumed directly by the gateway** — e.g. `buildGoogleCalendarClient`/`buildInMemoryCalendarClient`/`OAUTH_SECRET_LABEL` (`mount-open-cores.ts:60-75`), `EmailProjectCacheResolver`, `buildReminderStoreBackend`, `buildProductionResearchCoreWiring`; `open/composer.ts`, `gateway/http/app-focus-surface.ts`, `gateway/http/project-launcher-store.ts`, `app/lib/reminders-client.ts`, `chat-core/search.ts` also import core barrels (and one deep import: `open/composer.ts` → `cores/free/reminders/src/backend.ts`).
6. **Schedulers** — calendar pre-meeting-brief and email daily-triage schedulers are constructed from gateway wiring (`gateway/cores/calendar-wiring.ts`, `gateway/cores/email-managed-wiring.ts`) around core-exported scheduler factories; they self-tick like `reminders/tick.ts`.

## 5. Internal layering (per core)

manifest (constants + loader) → backend (domain contract + in-memory + production impls) → tools / mcp-tools-extra (capability-guarded MCP surface) → chat-commands (pure parser) → chat-bridge or gateway-side filter (envelope adapter) → ui (declarative metadata) → wiring-production (single-source composition helper, research/scraping only in practice). Sidecar cores add `migrations/` + a resolver (`research/src/store-resolver.ts`, `email/src/cache.ts:279`, `code-gen/src/sidecar/store.ts:353`; calendar's resolver lives gateway-side at `gateway/cores/calendar-wiring.ts:148`). This layering is genuinely consistent where it exists — the drift is in *which* layers each core has and *where* the filter adapter lives (§6.4).

## 6. Architectural debt

### 6.1 [P1] Cores bypass the package seam with `../../../../` imports into monorepo internals
Evidence in §4. The whole plugin story (SDK-CONTRACT.md; `bundled-registry.ts:34-40` "shape mirrors third-party Cores byte-for-byte") is falsified by 14 relative imports into `persistence/`, `runtime/`, `migrations/`, `connect/`, `onboarding/`. Consequences: cores are not publishable/relocatable; the Connect-dormant + tenant-rename refactors will break `agent-settings` invisibly; `research` silently depends on `runtime/models.ts` env-frozen constants (§7.3). **Sketch:** create the missing narrow seams — export `applyProjectScopedMigrations` from a small `@neutronai/migrations` surface (or ship a copy in cores-runtime, which already owns `data-namespace.ts`), move `AgentEngagementMode` + `TELEGRAM_BIND_TOKEN_TTL_MS` into an injected backend dep (agent-settings already injects `AgentProfileBackend`/`AgentSettingsTelegram`; the engagement-mode enum and TTL belong in `ToolDeps`), pass model ids into research the way email receives `emailModel: getBestModel` (`mount-open-cores.ts:288`), and use the already-declared `ProjectDb` type via `@neutronai/runtime` or a persistence workspace dep.

### 6.2 [P1] code-gen Core is ~70% production-dead legacy, superseded by Trident
`gateway/boot-helpers.ts:1128-1152`: "the codegen_core module now ONLY backs the four legacy `codegen_*` MCP tools; `/code <task>` no longer touches it… the Code-Gen Core's production WRAPPER was RETIRED… the codegen_* MCP tools dispatch into `buildSkeletonCodegenRunner`, whose `run(...)` throws `CodegenNotConfiguredError`." Zero non-self consumers of `buildCodegenWiring` (`src/wiring-production.ts:109`), `runtime-runner.ts` (481), `substrate-runtime.ts` (488), `host-runners.ts`, `worktree-resolver.ts`, `sidecar/store.ts` (534), `prompts/{forge,argus}-system.ts` — that's ~2,900 src + ~2,800 test lines exercising a pipeline that cannot run in production. `trident/code-command.ts:52` re-implements "the same grammar as the Code-Gen Core's `parseCodeCommand`" (grammar duplicated in two places), and code-gen's in-tree Forge/Argus prompts are a diverging fork of `trident/prompts.ts:102,288`. **Sketch:** as part of (or ahead of) the trident-v2 rearchitecture, shrink codegen-core to manifest + skeleton backend + tools (keeping the install_ok=TRUE + loud-error contract, §7.8) and delete the retired runtime stack + its tests, or move anything trident-v2 wants into `trident/`.

### 6.3 [P1] The real Core contract is undocumented duck-typing; the documented SDK contract is unused
`cores/sdk/SDK-CONTRACT.md` (475 lines) specifies `Connector`, `mountCoreRoutes`, JWT validation, `SecretsAccessor`, reconciliation — of which the nine bundled cores use only `parseManifest` + types (§4). The contract they actually build against is (a) the `install-bundled.ts` barrel duck-type (`buildTools`/`buildExtraTools`/`LAUNCHER_ICON`, `install-bundled.ts:751-784,854,289-294`), (b) `cores-runtime` `CapabilityGuard`/`SecretAuditLog`/`decideDataLayout`, (c) the ChatCommandFilter envelope, and (d) per-core `ToolDeps` shapes hand-maintained in `gateway/boot-helpers.ts:745-1200`. On top of that there are still **two parallel manifest validators** (`core-sdk/` install-time vs `cores/sdk/` runtime — SDK-CONTRACT.md §"Relationship to `core-sdk/`") with an intentional capability-enum divergence. **Sketch:** write the *actual* v1 bundled-Core contract into `cores/sdk` (exported `BundledCoreModule` interface: `buildTools`, optional `buildExtraTools`, `LAUNCHER_ICON`, `loadManifest`) and have `install-bundled.ts` type against it; fold or clearly quarantine the unused Connector/route/JWT half as "P3 third-party surface, no current consumer"; merge or generate one of the two manifest validators from the other.

### 6.4 [P2] Four different chat-command integration patterns
1. In-core filter factory: email `src/chat-bridge.ts:70`, research `src/chat-bridge.ts:56`, scraping `src/chat-bridge.ts` — each structurally re-declares the gateway `ChatCommandFilter` Input/Result/match contract (3 near-identical copies).
2. Gateway-side filter builders wrapping core parse/execute: reminders (`gateway/boot-helpers.ts:555`), calendar (`:694`).
3. Gateway-side router wrapper, not a filter: tasks (`gateway/cores/tasks-chat-router.ts:109 wrapWithTasksChatRouter`).
4. Grammar duplicated into another subsystem: `/code` → `trident/code-command.ts` (§6.2).
**Sketch:** pick pattern 1 (core owns `create<X>ChatCommandFilter`; gateway only chains) as canonical, export a shared `CoreChatCommandFilter` structural type from cores-sdk, and migrate reminders/calendar/tasks. Preserve chain order + fall-through-on-throw semantics exactly (§7.1).

### 6.5 [P1→P2] Per-project sidecar resolver copy-pasted 4×, and only research has the path-traversal guard
Same init-promise-dedup + handle-cache + `*SidecarMismatchError` + `closeAll()` pattern in `research/src/store-resolver.ts:90-200`, `email/src/cache.ts:279-345`, `code-gen/src/sidecar/store.ts:353-435`, and (simplified, gateway-side) `gateway/cores/calendar-wiring.ts:148-159`. Research alone adds `safeResolveProjectRoot` (`store-resolver.ts:128-159`) rejecting `..`/NUL/absolute `project_id` **before any FS op**, with an explicit docstring saying tool/chat inputs are external. Email (`cache.ts:296-298,327-333`) and code-gen and calendar do plain `join(owner_home,'Projects',project_id)` with no boundary check — the exact hole research documents. Single-owner installs soften the blast radius (the agent is the caller), but the inconsistent security posture is a defect. **Sketch:** extract one `ProjectSidecarResolver<H>` (open+migrate+meta-check+traversal-guard+dedup+closeAll) into `cores/runtime` (it already owns `data-namespace.ts`) parameterized by dir/db/meta-table/handle-factory; all four adopt it, inheriting the traversal guard.

### 6.6 [P2] `loadManifest`/manifest.ts boilerplate duplicated 9×
All nine `src/manifest.ts` files repeat the same ~40-line `defaultPackageJsonPath` + read + JSON-parse + `"neutron"`-block-extract + `parseManifest` (e.g. `tasks/src/manifest.ts:64-87`), which itself re-implements `cores/runtime/loader.ts:87-164 readCorePackage`. **Sketch:** `loadManifestFromPackageDir(import.meta.url)` helper in cores-sdk; per-core manifest.ts shrinks to constants.

### 6.7 [P2] God files inside cores
- `email/src/backend.ts` (2,003) — six clusters: GmailClient contract types (:51-315), typed errors (:317-385), in-memory client (:462-765), seeded in-memory client (:766-1075), Google REST client + MIME/base64/HTML helpers (:1076-1907, incl. `buildRawMessage` + header-injection guard :1434-1513), EmailSummarizer (:1908-2003). ### 6.8 [P2] Duplicated NL date/time parsing between reminders and calendar
`reminders/src/chat-commands.ts:377-649` (`resolveTimeSpec`, `matchWeekday`, `nextWeekdayOccurrence`, `nextMonthlyOccurrence`) vs `calendar/src/chat-commands.ts:93,166-210,263,315-338` (WEEKDAYS map, today/tomorrow/weekday@HH:mm, `Nm/Nh` relative). ## 7. Load-bearing subtleties (a behavior-preserving refactor must NOT break)

1. **Chat filter chain semantics** (`gateway/boot-helpers.ts:519-543`): first non-null result wins; a *throwing* filter logs and **falls through to the next** (never poisons the chain, never surfaces the error). Open chain order calendar→email→reminders→research (`mount-open-cores.ts:310-329`). Reordering or letting exceptions propagate changes which Core claims ambiguous input and whether the LLM fallback fires.
2. **Optional-until-credentialed** (`mount-open-cores.ts:28-35,177-243`): no `NEUTRON_CORES_GOOGLE_CLIENT_ID` ⇒ in-memory calendar/gmail clients ⇒ `/cal`//`/email` answer with a graceful empty result — "never a hard error, never a boot block". Substrate `null` ⇒ research filter still claims `/research` but reports reconnect failure; email triage LLM stub **throws by design** so triage renders its deterministic structured-row fallback (`:272-277`, `substrate-llm.ts` header: "a transient Haiku outage never silently drops a daily triage").
3. **Per-call model thunks** for email (`getBestModel` accessor resolved at match/dispatch time — `mount-open-cores.ts:156-159,286-288,319-321`); freezing to a constant during refactor silently breaks the model-watchdog flip. Conversely research's frozen constants are the *current* behavior — "fixing" it per §6.10 is a functional change to schedule deliberately.
4. **One backend instance shared by MCP tools AND chat filter** ("agent-native parity", `mount-open-cores.ts:23-27,223-268`): the same `calendarClient`, the same `EmailProjectCacheResolver`, the same research `project_backend`. Splitting instances forks state (separate sidecar handles, separate caches).
5. **Research web-fetch SSRF enforcement is Core-side by contract**: `network:browse` promises an allow-list *the runtime does NOT enforce* (SDK-CONTRACT.md §manifest, 2026-05-20 note). `research/src/web-fetch.ts` is the reference enforcer: unconditional RFC-1918/loopback/link-local/metadata/`file:`/`data:` rejects, redirect-refusal, 5 MB cap, content-type sniffing, and a **URL-rewrite DNS-pinning approach chosen because Bun's `undici.Agent` is a stub** (`web-fetch.ts:287-292`). Swapping to the "canonical" undici dispatcher pattern breaks under Bun.
6. **Sidecar meta checks fail loud before returning a handle** (`ResearchSidecarMismatchError` `store-resolver.ts:239`, `EmailSidecarMismatchError` `cache.ts:353`, `CalendarSidecarMismatchError` `cache.ts:218`, codegen `store.ts:435`): a sidecar copied between projects must never silently serve the wrong project's data. Research's traversal guard must run **before any FS op** (`store-resolver.ts:119-127`).
7. **Init-promise dedup** in every resolver (`store-resolver.ts:186-200` et al.): concurrent first-resolves await one init; the pending map is cleared in `finally`. Naive "simplification" to a plain cache introduces double-open/migration races.
8. **codegen skeleton contract** (`boot-helpers.ts:1125-1152`, `code-gen/src/backend.ts:405-428`): `install_ok` must stay TRUE while every dispatch throws typed `CodegenNotConfiguredError` — the fall-through is deliberately observable; deleting the skeleton or making install fail changes install-state rows and tool-error surfaces.
9. **Ordering invariants**: email list/search are newest-first by `internalDate` DESC; calendar events ascending (`email/src/backend.ts:24-27` documents the asymmetry). Tasks' `tasks_pick_next` gates on `read:` not `write:` capability (`tasks/AGENTS.md`).
10. **Email is draft-only by design**: `gmail.send` excluded from client surface, capabilities, and OAuth scopes (`email/src/backend.ts:29-37`); a refactor that "completes" the client adds a capability the manifest doesn't grant.
11. **Boot determinism**: bundled-registry ordering is full-path lexicographic (`loader.ts:263-267`); duplicate-slug semantics differ same-root (throw) vs cross-root (precedence + telemetry, `bundled-registry.ts:14-23`); a bad manifest blocks boot (`:42-47`). `LAUNCHER_ICON` module-load failures must degrade per `install-bundled.ts:289-294` rules.
12. **agent-settings honest no-ops**: absent composer-threaded backends must report `available:false` failure, not fake success (`boot-helpers.ts:1163-1180`, Argus r5 note).

## 8. Test posture

Strong per-core unit coverage overall (test:src ratio ≈ 0.75; email 13 files/4,013 lines incl. a 1,141-line backend suite; research 12 files incl. an 885-line web-fetch SSRF suite). Tests are hermetic: in-memory clients, `mkdtempSync` sidecars, injected fakes — no network, no PGLite (so the known PGLite boot flake doesn't touch this subsystem). Install-lifecycle tests are copy-pasted per core (identical preambles across reminders/calendar/email/research/code-gen). Gateway-side composition is well covered too (`gateway/__tests__/*-production-composer.test.ts` deliberately route through the same `wiring-production` factories as prod — the PR #256 lesson recorded in `research/src/wiring-production.ts:12-19`). Gaps: **code-gen's biggest suites test production-dead code** (§6.2 — `substrate-runtime.test.ts` 602, `runtime-runner.test.ts` 314 exercise the retired pipeline: green tests, zero prod meaning); no cross-core contract test asserts every bundled barrel satisfies the `install-bundled` duck-type (a core dropping `buildTools` is only caught at boot / in gateway tests); no traversal-guard tests exist for email/code-gen/calendar resolvers because the guards don't exist. Flake risk: low; scheduler tests use injected clocks.

## 9. What the refactor should do here

1. **Define the real Core contract in one place** (§6.3): a typed `BundledCoreModule` in cores-sdk; type `install-bundled.ts` against it; document it in SDK-CONTRACT.md and quarantine the unused Connector/JWT/route surface as future third-party API. This is the highest-leverage move — everything else becomes mechanical against a named contract.
2. **Cut the 14 `../../../../` imports** (§6.1) by adding the three missing seams (migrations applier, model accessor injection, engagement-mode/TTL injection into agent-settings deps). This also de-risks the Connect-dormant and tenant-rename refactors.
3. **Extract shared `ProjectSidecarResolver`** into cores-runtime with research's traversal guard as the default (§6.5) — closes a real inconsistency while deleting ~600 duplicated lines. Keep mismatch-error codes and init-dedup semantics byte-compatible.
4. **Standardize chat-command integration on the in-core filter-factory pattern** (§6.4); keep chain order and throw-fall-through exactly (§7.1).
5. **Shrink code-gen to its live surface** in coordination with trident-v2 (§6.2); until then mark the retired modules clearly.
6. **Delete dead matter now** (zero behavior risk): `cores/free/notes/`, `cores/free/reminders/manifest.json`, stale `cores/AGENTS.md` core list.
